01

Mission brief

The friction is operational

Good tools are not enough. People need a usable route through them.

Open technologies can help people communicate, transact, preserve information, and coordinate with greater autonomy. Yet each tool arrives with its own interface, vocabulary, setup, and security assumptions.

Freedom Skills packages that hard-won knowledge into portable, agent-ready workflows. A compatible agent can recognize the task, load the relevant instructions, use the right tools, and surface the decisions that still belong to the human.

✳

Design constraint
Convenience should not require centralized, data-extracting services.

02

Operating procedure

Portable by construction

One small file opens the right field kit.

An Agent Skill is a folder built around SKILL.md. Metadata helps compatible agents discover it; detailed instructions load only when the task calls for them.

  1. A

    Place

    Copy or clone a skill folder into the compatible agent’s skills directory.

    skills/<skill-name>/
  2. B

    Discover

    The agent reads compact metadata to know when the skill is relevant.

    name + description
  3. C

    Load

    When triggered, it reads the full workflow and only the resources the job needs.

    SKILL.md → references/
  4. D

    Operate

    Scripts, checks, guardrails, and user confirmations guide real execution.

    act → verify → report

04

Safety protocol

Human remains in command

Instructions improve judgment. They do not erase risk.

Freedom technologies often handle identity, money, private communication, and public records. Every workflow should make its operating boundary legible.

Review the source ↗
□

Inspect before installRead the skill, bundled scripts, prerequisites, and network behavior.

□

Keep secrets server-sideNever commit mnemonics, access tokens, private keys, or account credentials.

□

Confirm irreversible actionSending money, publishing, messaging, and changing accounts warrant explicit human intent.

□

Verify the resultCheck exit codes, receipts, remote state, and timestamps. A command running is not proof of success.
